## 의도부터 본다

OmO는 문장을 글자 그대로만 보지 않습니다. 사용자가 지금 설명을 원하는지, 조사를 원하는지, 바로 고치길 원하는지 먼저 판단합니다.

| 사용자가 말한 것 | OmO가 해야 하는 일 |
|---|---|
| “왜 깨졌어?” | 재현하고, 원인을 찾고, 가능하면 고칩니다. |
| “한번 봐줄래?” | 요약에서 멈추지 않고 조사와 해결까지 갑니다. |
| “가장 좋은 방법은?” | 방법을 고르고, 코드 맥락이 있으면 실행합니다. |
| “리뷰해줘” | 요약보다 버그, 위험, 검증 공백을 먼저 찾습니다. |

## 계획과 실행을 나눈다

복잡한 작업은 “생각하는 역할”과 “고치는 역할”을 분리합니다.

| Agent | 맡는 일 |
|---|---|
| Prometheus | 질문하고 조사해서 실행 계획을 씁니다. production code는 건드리지 않습니다. |
| Metis | 계획이 굳기 전에 숨은 의도와 빠진 제약을 찾습니다. |
| Momus | 파일, acceptance criteria, 검증 방법이 흐리면 계획을 돌려보냅니다. |
| Atlas | 승인된 계획을 읽고 작업 단위로 실행합니다. |
| Sisyphus-Junior | category로 생성되는 focused executor. 하나의 단위만 맡습니다. |

좋은 계획은 구현자에게 “이건 어떻게 할까요?”를 남기지 않습니다.

## 모델은 역할별로 쓴다

모든 일을 가장 비싼 모델에 던지지 않습니다. 검색은 빠르게, 아키텍처는 깊게, UI는 시각 추론이 강한 쪽으로 보냅니다.

| 계열 | 적합한 일 |
|---|---|
| Claude-like | Sisyphus식 orchestration, 긴 규칙 준수, 구조화된 실행 |
| GPT | Hephaestus, Oracle, Momus처럼 명시적 추론이 중요한 일 |
| Gemini | visual/frontend, 디자인, 큰 맥락의 시각적 판단 |
| Fast utility | Explore, Librarian, quick fix, grep식 조사 |

## Category + Skill

Category는 “무슨 일인가”입니다. Skill은 “그 일을 어떻게 할 것인가”입니다.

| Category | 용도 |
|---|---|
| `visual-engineering` | Frontend, UI/UX, design, styling, animation |
| `ultrabrain` | hard logic, architecture, deep reasoning |
| `deep` | 자율 조사와 end-to-end 구현 |
| `artistry` | 창의적이거나 비정형적인 문제 |
| `quick` | 단일 파일 수정, typo, 작은 변경 |
| `writing` | 문서와 prose |

```ts
task({
  category: "visual-engineering",
  load_skills: ["frontend-ui-ux", "playwright"],
  run_in_background: false,
  prompt: "Implement and browser-verify the responsive sidebar.",
});
```

## 계속 밀어 올린다

Sisyphus라는 이름은 장식이 아닙니다. 일이 끝나지 않았으면 다시 이어갑니다.

| 장치 | 막는 실패 |
|---|---|
| Todo continuation | todo가 남았는데 agent가 멈추는 상황 |
| Ralph loop / ulw loop | 긴 작업이 여러 번의 continuation을 필요로 하는 상황 |
| Boulder state | `/start-work`가 계획 실행을 세션 너머로 이어가야 하는 상황 |
| Session recovery | tool result 누락, empty content, context limit 문제 |
| Runtime fallback | provider/model 실패 후 다음 fallback으로 넘어가야 하는 상황 |
