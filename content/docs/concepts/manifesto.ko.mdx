OmO는 의견이 강한 도구입니다. 기능들은 작은 원칙 집합에서 따라 나옵니다. 이 페이지는 그 원칙입니다 — 기능 카탈로그가 아니라.

## 인간 개입은 실패 신호

자율주행을 생각해보세요. 사람이 운전대를 잡아야 한다면 그건 기능이 아니라 시스템이 실패한 것입니다. 차가 그 상황을 스스로 처리하지 못한 거죠.

코딩도 다르지 않습니다. AI의 반쪽짜리 코드를 고치고, 뻔한 실수를 일일이 수정하고, 한 걸음씩 손잡고 이끌고, 같은 요구사항을 반복해서 설명하고 있다면 — 그건 협업이 아닙니다. AI가 자기 일을 못 하고 있는 겁니다.

OmO는 개입이 예외가 되도록 설계되어 있습니다. 루프가 아니라.

## 구분되지 않는 코드

에이전트가 쓴 코드는 시니어 엔지니어가 쓴 코드와 구분되지 않아야 합니다. "정리가 필요한 AI 코드"가 아니라, "그럭저럭 출발점"이 아니라, 실제 최종 프로덕션 코드.

실천적으로:

- 기존 코드베이스 패턴을 그대로 따른다.
- 시키지 않아도 에러 처리를 한다.
- 테스트는 실제로 필요한 걸 테스트한다.
- AI slop 없음 — 과잉 설계 없음, scope creep 없음, 장식적 추상화 없음.
- 댓글은 가치가 있을 때만 쓴다.

커밋이 사람 작업인지 에이전트 작업인지 구분할 수 있다면 에이전트가 실패한 것입니다.

## 토큰 비용 vs 생산성

생산성이 오르면 토큰 사용량이 늘어나는 건 괜찮습니다. 병렬 리서치, 개입 없는 end-to-end 완료, 철저한 검증, 세션 간 지식 축적 — 다 가치 있는 지출입니다.

불필요한 낭비는 아닙니다. 시스템은 다음을 최적화합니다.

- 유틸리티 레인에는 저렴한 모델 (Haiku, Flash, GPT-5.4 Mini).
- 컨텍스트가 충분하면 추가 탐색 없음.
- 세션 간 학습 캐싱.
- 충분할 때 멈추기.

토큰 효율은 중요합니다. 단, 작업 품질이나 사용자 인지 부담을 희생해서까지는 아닙니다.

## 인지 부담 최소화

사용자의 역할은 원하는 걸 말하는 것입니다. 그 외 모두는 에이전트의 일.

같은 결과로 가는 두 경로:

- **Prometheus 모드** — "auth를 추가하고 싶어"라고 말합니다. Prometheus가 코드베이스를 조사하고, 실제 발견을 바탕으로 명확화 질문을 던지고, 못 보던 엣지 케이스를 들춰내고, 계획을 씁니다. 의도는 사용자가, 구조는 에이전트가 제공합니다.
- **Ultrawork 모드** — `ulw add authentication`이라고 칩니다. 에이전트가 접근 방식을 정하고, 베스트 프랙티스를 리서치하고, 컨벤션을 따라 구현하고, 끝날 때까지 계속합니다. 의도는 사용자가, 나머지는 에이전트가.

사용자는 무엇을 원하는지 표현할 뿐, 어떻게 끝낼지를 관리하지 않습니다.

## 예측 가능, 지속, 위임 가능

이상적인 에이전트는 컴파일러처럼 동작합니다. 마크다운이 들어가면 동작하는 코드가 나옵니다.

- **예측 가능.** 같은 코드베이스 패턴, 같은 요구사항, 같은 제약 → 일관된 결과. 랜덤도 아니고, 놀랍지도 않고, 아무도 부탁하지 않은 "창의성"도 없습니다.
- **지속.** 작업이 중단을 견딥니다. 세션이 죽어도 `/start-work`로 재개됩니다. 다중일 프로젝트도 컨텍스트가 보존됩니다. 에이전트가 상태를 들고 있고, 사용자는 아닙니다.
- **위임 가능.** 명확한 acceptance criteria, 독립 검증. 실패 시 자기 교정. 정말 필요할 때만 escalation. "거의 끝남"이 아니라 완료.

## 핵심 루프

```text
Human Intent  →  Agent Execution  →  Verified Result
      ↑                                      ↓
      └──────────────── Minimum ─────────────┘
        (정말 실패할 때만 개입)
```

OmO의 표면적은 이 루프 위에 매핑됩니다.

| 기능 | 목적 |
|---|---|
| Prometheus | 인터뷰로 의도 추출. |
| Metis | 계획이 굳기 전 모호함 잡기. |
| Momus | 계획이 완전한지 검증. |
| Sisyphus 오케스트레이터 | 마이크로 매니지 없이 작업 조율. |
| Todo continuation | "끝났다"라는 거짓말을 막고 완료 강제. |
| Category 시스템 | 사람 결정 없이 최적 모델로 라우팅. |
| Background agents | 사용자를 막지 않는 병렬 리서치. |
| Wisdom accumulation | 작업에서 배우고, 실수를 반복하지 않기. |

## 실천적으로

다음을 할 수 있어야 합니다.

1. 원하는 걸 원하는 디테일로 묘사.
2. 범위가 흐릿하면 에이전트가 인터뷰하게 두기.
3. 계획 확인 또는 그냥 `ulw`로 직행.
4. 자리 비우기.
5. 돌아왔을 때 완료된, 검증된, 프로덕션 코드가 있기.

이게 안 된다면 시스템 어딘가가 개선되어야 합니다.

## 우리가 짓고 있는 미래

개발자가 "AI가 어떻게 빌드하게 만들지"가 아니라 "무엇을 빌드할지"에 집중하는 세계. 누가/무엇이 썼는지와 무관하게 코드 품질이 보장되는 세계. 복잡한 프로젝트가 단순한 프로젝트만큼 다루기 쉬운 세계(다만 오래 걸릴 뿐). "prompt engineering"이 "compiler debugging"만큼 구식이 되는 세계.

에이전트는 보이지 않아야 합니다 — 숨겨졌다는 의미가 아니라, 아무도 의식할 필요가 없을 정도로 신뢰할 수 있다는 의미로. 전기처럼. 스위치를 켜면 불이 들어옵니다. 송전망은 생각하지 않죠.
