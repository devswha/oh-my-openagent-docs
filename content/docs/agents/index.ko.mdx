## 핵심 순서

OpenCode에서 core agent의 Tab 순서는 고정입니다. Sisyphus, Hephaestus, Prometheus, Atlas가 먼저 나오고 나머지가 뒤따릅니다.

| Agent | 역할 | 메모 |
|---|---|---|
| Sisyphus | 메인 orchestrator | 의도 파악, 위임, todo 추적, 완료까지 진행 |
| Hephaestus | GPT-native deep worker | 아키텍처, 어려운 디버깅, cross-domain 판단 |
| Prometheus | planner/interviewer | 질문하고 조사해서 계획 작성. production code는 수정하지 않음 |
| Atlas | plan executor | Prometheus plan을 읽고 boulder state로 이어서 실행 |
| Oracle | read-only consultant | 낯선 구조, 보안/성능 tradeoff, hard debugging |
| Librarian | docs/remote repo researcher | 공식 문서, OSS 예제, 외부 repo 조사 |
| Explore | codebase searcher | “X가 어디 있지?”를 빠르게 찾음 |
| Multimodal-Looker | media reader | 이미지, PDF, diagram, screenshot 분석 |
| Metis | pre-plan gap finder | 숨은 의도와 빠진 제약 찾기 |
| Momus | plan/result reviewer | 모호하거나 검증 불가능한 계획 거절 |
| Sisyphus-Junior | focused executor | category로 생성되어 하나의 단위만 처리 |

## 도구 경계

똑똑한 agent라도 아무 일이나 하면 위험합니다. OmO는 역할별로 경계를 둡니다.

| Agent | 못 하는 일 |
|---|---|
| Oracle | write/edit/delegation 없음 |
| Librarian | write/edit/delegation 없음 |
| Explore | write/edit/delegation 없음 |
| Multimodal-Looker | media analysis 중심 |
| Atlas | 직접 구현보다 계획 실행 orchestration |
| Sisyphus-Junior | 재위임 없음. 맡은 단위만 구현하고 검증 |

## Category Worker

`task` tool에 `category`를 주면 해당 domain에 맞춘 Sisyphus-Junior가 뜹니다.

| Category | 맡기기 좋은 일 |
|---|---|
| `visual-engineering` | Frontend/UI/design. 작은 CSS라도 여기로 보냅니다. |
| `ultrabrain` | hard reasoning과 architecture |
| `deep` | 조사와 구현이 같이 필요한 큰 작업 |
| `artistry` | 정석이 아닌 접근이 필요한 일 |
| `quick` | 작은 변경, typo, 단일 파일 수정 |
| `writing` | 문서와 prose |

## Background Agents

Explore나 Librarian은 background로 띄워두면 좋습니다. 메인 세션은 구현을 계속하고, 조사 결과는 나중에 받습니다.

```ts
task({
  subagent_type: "explore",
  load_skills: [],
  run_in_background: true,
  prompt: "Find all auth policy implementations and summarize file paths.",
});
```

다음 구현 결정을 막는 질문이면 synchronous mode로 기다립니다. 단순 조사는 background가 낫습니다.
