OmO에는 여러 진입점이 있습니다. 보통은 그중 하나만 쓰면 됩니다. 이 페이지는 언제 어느 길로 가야 하는지 빠르게 정리한 지도입니다.

## 어떤 워크플로우를 쓸까

<Cards>
  <Card title="일반 프롬프트">작은 수정이나 직접 질문. 가장 가벼운 길 — Sisyphus와 그냥 대화합니다.</Card>
  <Card title="Ultrawork (`ulw`)">복잡하지만 자율성을 맡기고 싶을 때. Sisyphus가 탐색, 위임, 구현, 검증, 이어가기까지 합니다.</Card>
  <Card title="Prometheus + Atlas">복잡하고 정확도가 중요할 때. `@plan`으로 먼저 잠그고, `/start-work`로 Atlas가 받아서 실행합니다.</Card>
  <Card title="Hephaestus">GPT-native deep reasoning. 아키텍처, 디버깅, 합성 판단을 한 명이 자율로 처리합니다.</Card>
  <Card title="Category task">좁은 구현 단위 하나. category와 skill을 단 focused Sisyphus-Junior를 띄웁니다.</Card>

## Ultrawork

`ulw`는 "그냥 끝까지 처리해줘" 경로입니다. 목표를 주고, 가능하면 동작을 증명할 표면도 같이 주세요.

```bash
ulw fix the login error message and verify success/failure in the browser
```

좋은 `ulw` 프롬프트는 세 가지를 짚습니다.

- 사용자가 보게 될 동작.
- 그 동작을 증명할 표면: browser, CLI, API, tests, docs build.
- 절대 건드리면 안 되는 경계.

## Prometheus + Atlas

속도보다 정확도가 더 중요할 때 Prometheus를 부릅니다.

<Steps>

<Step>

### Prometheus로 계획 잠그기

Sisyphus 안에서 `@plan`을 입력하거나, Tab으로 Prometheus로 전환하세요.

```bash
@plan refactor the auth module without behavior changes
```

요구사항이 또렷해질 때까지 인터뷰 질문에 답하세요. Prometheus가 `.sisyphus/plans/` 아래에 계획을 씁니다.

</Step>

<Step>

### Atlas에 넘기기

```bash
/start-work
```

Atlas는 계획을 읽고, boulder state를 만들고, 실행 단위를 위임합니다. 세션이 끊겨도 나중에 이어서 재개할 수 있습니다.

</Step>

<Step>

### 필요하면 복구

`/start-work`가 active plan을 못 찾는다고 하면, Prometheus 계획을 먼저 만들거나 stale `.sisyphus/boulder.json`을 정리하세요.

</Step>

</Steps>

## Hephaestus vs Sisyphus

| 기준 | Hephaestus | Sisyphus + `ulw` |
|---|---|---|
| 모델 스타일 | GPT-native 자율 deep worker. | category와 skill로 위임하는 오케스트레이터. |
| 잘 맞는 일 | 아키텍처, 어려운 디버깅, cross-domain 추론. | 대부분의 "그냥 처리해줘" 복잡 작업. |
| 계획 | 실행하면서 자체 계획. | Prometheus 계획이나 자율 탐색 사용. |
| 위임 | 맥락은 Explore, Librarian에 의존. | category 워커와 specialist 활용. |

기본은 `ulw`로 두세요. GPT-5.4 식 깊은 추론이 콕 집어 필요할 때만 Hephaestus로 가세요.

## 에이전트에게 설치를 맡길 때

upstream README도 에이전트가 OmO를 설치하는 흐름을 권장합니다. 설치를 맡은 에이전트는 이렇게 움직여야 합니다.

1. 어떤 구독과 provider가 있는지 묻습니다.
2. 답에 맞춰 `bunx oh-my-openagent install --no-tui ...`를 실행합니다.
3. 필요하면 인터랙티브 터미널에서 `opencode auth login`을 안내합니다.
4. `bunx oh-my-openagent doctor`를 실행합니다.
5. 모델 setup과 안전한 override를 설명합니다.

에이전트가 모델 설정이나 hook/MCP를 사용자 동의 없이 조용히 바꾸면 안 됩니다.

## 모델 선택의 감

- Sisyphus는 Claude Opus, Kimi K2.5, GLM 5 스타일 모델에서 가장 안정적입니다.
- Hephaestus, Oracle, Momus는 GPT-friendly 또는 GPT-native입니다.
- Visual engineering은 Gemini 식 frontend reasoning이 잘 맞습니다.
- Explore와 Librarian은 빠르고 저렴해야 합니다.
- `fallback_models`와 runtime fallback은 워크플로우를 멈추지 않고 실패를 넘깁니다.
