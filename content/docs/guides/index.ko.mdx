## 어디로 시작할까

| 상황 | 쓰면 좋은 것 | 이유 |
|---|---|---|
| 작은 수정이나 질문 | 일반 prompt | 가장 빠릅니다. |
| 복잡하지만 맡기고 싶을 때 | `ulw` / `ultrawork` | Sisyphus가 조사, 위임, 구현, 검증까지 갑니다. |
| 정확한 계획이 먼저일 때 | Prometheus `@plan` → `/start-work` | 구현 전에 결정을 끝냅니다. |
| 깊은 추론이 필요할 때 | Hephaestus | 아키텍처, 어려운 디버깅, cross-domain 판단에 맞습니다. |
| 좁은 구현 단위 하나 | `task(category=..., load_skills=[...])` | 맞는 category worker에게 넘깁니다. |

## Ultrawork

`ulw`는 “그냥 끝까지 해줘”에 가깝습니다. 목표와 검증 표면을 같이 주면 좋습니다.

```text
ulw fix the login error message and verify success/failure in the browser
```

좋은 prompt는 짧아도 됩니다.

- 사용자가 보게 될 동작
- 확인해야 할 표면: browser, CLI, API, tests, docs build
- 건드리면 안 되는 경계

## Prometheus + Atlas

속도보다 정확도가 중요하면 이 경로를 탑니다.

1. Sisyphus에서 `@plan "..."`을 입력하거나 Tab으로 Prometheus를 고릅니다.
2. 요구사항이 또렷해질 때까지 질문에 답합니다.
3. Prometheus가 `.sisyphus/plans/` 아래에 계획을 씁니다.
4. `/start-work`를 실행합니다.
5. Atlas가 계획을 읽고 작업 단위로 실행합니다.

`/start-work`가 active plan을 못 찾으면 Prometheus plan부터 만들거나 stale `.sisyphus/boulder.json`을 정리하세요.

## Hephaestus vs Sisyphus

| 기준 | Hephaestus | Sisyphus + `ulw` |
|---|---|---|
| 스타일 | GPT-native deep worker | category와 skill로 위임하는 orchestrator |
| 잘하는 일 | 아키텍처, 어려운 debugging, 합성 판단 | 대부분의 “맡아서 처리해줘” 작업 |
| 계획 | 실행하면서 자체 계획 | Prometheus plan 또는 자율 탐색 |
| 조사 | Explore/Librarian으로 맥락을 많이 모음 | category worker와 specialist를 섞음 |

대부분은 `ulw`로 충분합니다. GPT식 깊은 판단이 필요할 때 Hephaestus를 고르세요.

## 설치를 에이전트에게 맡길 때

upstream README도 agent-driven install을 전제로 설명합니다. 설치를 맡기는 agent는 보통 이렇게 움직입니다.

1. 사용자가 가진 provider/subscription을 묻습니다.
2. 답에 맞춰 `bunx oh-my-opencode install --no-tui ...`를 실행합니다.
3. 필요하면 interactive terminal에서 `opencode auth login`을 안내합니다.
4. `bunx oh-my-opencode doctor`를 실행합니다.
5. model setup과 안전한 override만 설명합니다.

사용자가 말하지 않았다면 model 설정이나 hook/MCP를 마음대로 끄지 않습니다.

## 모델 선택 원칙

- Sisyphus는 Claude Opus, Kimi K2.5, GLM 5 계열에서 안정적입니다.
- Hephaestus, Oracle, Momus는 GPT-friendly 또는 GPT-native입니다.
- Visual engineering은 Gemini식 시각/frontend reasoning이 잘 맞습니다.
- Explore와 Librarian은 빠르고 저렴해야 합니다.
- `fallback_models`는 실패했을 때 workflow를 멈추지 않기 위한 안전망입니다.
