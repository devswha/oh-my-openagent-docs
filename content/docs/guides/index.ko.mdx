## 어떤 모드를 쓸까

| 상황 | 사용 | 이유 |
|---|---|---|
| 작은 수정 | 일반 프롬프트 | Sisyphus가 직접 처리하는 편이 빠릅니다. |
| 복잡하지만 맡기고 싶음 | `ulw` 또는 `ultrawork` | 탐색, 계획, 병렬 위임, 검증을 자동으로 이어갑니다. |
| 복잡하고 정확해야 함 | Prometheus → `/start-work` | 인터뷰 기반 계획과 Atlas 실행을 분리합니다. |
| 브라우저/CLI 동작 확인 필요 | 해당 surface 도구 포함 | Playwright, tmux, HTTP 요청으로 실제 사용 표면을 검증합니다. |

## Ultrawork 패턴

`ulw`는 “세 글자만 입력하고 떠나라”에 가까운 모드입니다. 코드베이스를 스캔하고, Explore/Librarian 같은 백그라운드 에이전트를 띄우고, 구현 후 검증까지 이어갑니다.

```text
ulw 결제 파서의 누락된 테스트 커버리지를 추가하고 실제 CLI 흐름까지 검증해줘
```

좋은 `ulw` 요청은 목표와 완료 표면을 함께 줍니다.

| 나쁜 요청 | 더 나은 요청 |
|---|---|
| “로그인 고쳐줘” | “로그인 실패 시 에러 메시지가 안 보이는 문제를 고치고 브라우저에서 실패/성공 흐름을 검증해줘” |
| “문서 정리” | “설치 문서의 provider 플래그와 doctor 검증 절차를 최신 CLI 기준으로 고쳐줘” |

## Prometheus + Atlas 패턴

정확성이 중요한 작업은 계획과 실행을 나눕니다.

<Steps>

### Prometheus에게 설명

`@plan` 또는 Prometheus agent로 전환해 목표, 금지사항, 검증 기준을 설명합니다.

### Metis와 Momus로 계획 검토

Metis는 빠진 요구사항을 찾고, Momus는 계획이 실행 가능한지 검토합니다.

### Atlas 실행

`/start-work`를 실행하면 Atlas가 계획을 읽고 Sisyphus-Junior와 specialist에게 작업을 분배합니다.

### 독립 검증

Atlas는 하위 에이전트의 “완료”를 그대로 믿지 않고, 직접 진단·빌드·테스트·수동 QA 결과를 확인합니다.

</Steps>

## 모델 설정 원칙

모델명은 계속 바뀝니다. 더 오래 가는 원칙은 아래입니다.

- Sisyphus/Prometheus: Claude 계열 또는 Claude-like 모델이 안정적입니다.
- Hephaestus/Oracle: GPT 계열 고성능 reasoning 모델이 어울립니다.
- Explore/Librarian: 빠르고 저렴한 검색 모델이 낫습니다.
- Visual 작업: `visual-engineering` 카테고리와 UI/UX 스킬을 같이 씁니다.

이유 없이 모든 agent를 최고가 모델로 올리면 비용만 늘고 속도는 떨어질 수 있습니다.
